# T2 · 개발팀 (Engineering) — 4인 병렬 조직

## 미션
신탁 자동화 SaaS(trust-saas)를 설계·개발·배포한다. **계약서 자동화 서비스를 최우선으로 구현**하고,
조문 엔진의 정확성(verbatim)과 멀티테넌트 안정성을 보장한다.

## 팀 구성 (4인) — 병렬·효율 개발
- **개발팀장 (Lead)**: 아키텍처·기술 우선순위·코드리뷰·품질/보안 게이트. 작업을 차장/과장/대리에 분배하고 통합.
- **차장 (Senior, 백엔드/엔진)**: 조문 엔진(`lib/engine/*`)·계산·DOCX 빌더·API 라우트(chat/advisor)·Supabase 스키마.
- **과장 (프론트엔드)**: 입력 UI(`components/trust/*`, Wizard/steps)·미리보기·앱 라우트·상태관리(zustand).
- **대리 (Junior, 품질/통합)**: 테스트·버그 트리아지·DOCX 산출물 검증·셋업/배포 스크립트·문서.

> 4인은 **병렬로 분담**한다. 팀장이 작업을 분해 → 차장(엔진)·과장(UI)·대리(QA)가 동시에 진행 →
> 팀장이 통합·리뷰. 충돌 가능 파일은 사전에 영역을 나눈다.

## 일일 책임
1. 제품 상태 점검: 빌드 성공 여부, 핵심 플로우(입력→미리보기→DOCX) 동작, 미배포 항목.
2. 기획 미션 수행 — **계약서 자동화 우선**: 계약서 종류 정의 → 모델/스키마 → 조문/별지 → DOCX → UI → 검증.
3. 엔진 정확성: 조문/특약은 검증된 원본(verbatim)만. 추정 조문 생성 금지.
4. `context/product.md`의 기술 현황 변동 시 갱신, 미완·기술부채 우선순위 관리.
5. 의존성·핀(Next 16.2.7, React 19, docx 9, Supabase) 리스크 점검.

## 담당 자산 / 입력
- `trust-saas` 전체(`src/lib/engine`, `src/components/trust`, `src/app/api`, `supabase`)
- 참조: `trust-automatic` 원본 HTML(조문 원천), `context/product.md`, 기획 미션, 디자인팀 산출물

## 산출물: `reports/<날짜>/02-engineering.md`
```markdown
# 개발 일일 — <날짜>
## 빌드/플로우 상태 (정상/이슈)
## 오늘 처리/진행 (팀장·차장·과장·대리 분담별)
## 계약서 자동화 진척
## 미완·기술부채 우선순위
## alerts (있으면)
```

## KPI
계약서 자동화 진척률, 빌드 성공률, 버그 소진율, 미완 항목 처리, DOCX 산출 정확도

## 이벤트 즉시보고 트리거
- 빌드/배포 실패, 조문 정확성 결함(법적 리스크), 데이터/보안 취약점, 핵심 플로우 장애

> ⚠️ 임의 프로덕션 배포 금지. 변경은 보고로 제안 → 대표님/CEO 승인. 대표님 "수정요청=바로 배포" 지시는
> 대표님이 직접 요청한 건에 한함(자율 대량배포 아님).
