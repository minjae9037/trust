## 1. 오늘 회사 우선순위 (1~3)

대표님 의사결정(inbox #3 "계약서 자동화 서비스를 가장 먼저 구현")을 최상위로 두고,
"1차 목표 = 계약서 자동화 서비스 구현·출시"에 자원을 집중한다.

### P1. 계약서 자동화 서비스 1차 — 현행 엔진 동작 검증 ★최우선
현재 엔진(`src/lib/engine/*`)은 담보신탁 7서류군 + 공동사업표준협약(2종군) verbatim 조문이
이식되어 있으나 **입력→미리보기→DOCX 산출 end-to-end 동작이 미검증**(KPI: "검증 전").
출시 로드맵의 모든 후속 작업이 이 검증 결과에 의존하므로 오늘의 1번.

### P2. 1차 자동화 대상 계약서 종류·필수조항·필드 정의서 확정
"계약서 자동화"의 범위(어떤 계약서를, 어떤 조항/필드로)를 사업팀(신탁 전문)이 정의해야
개발팀이 모델/스키마/조문을 확장할 수 있다. P1 검증과 병렬 진행 가능.

### P3. 운영 기반 점검 + 출시 메시지 골격
`.env.local`/로컬 구동 확인(개발), 포지셔닝·세일즈 원페이저 초안(마케팅)으로
출시 트랙을 미리 깐다. 단, **배포 환경·브랜드명은 대표님 결정 대기**라 실배포는 보류.

## 2. 팀별 배정 미션 (측정가능·실행가능)

### 개발팀(T2 · 4인 분담)
1. **[P0] 현행 엔진 end-to-end 동작 검증** — 담보신탁 7종 + 공동사업협약에 대해
   샘플 입력 1세트로 위저드 입력 → HTML 미리보기 → `.docx` 다운로드까지 성공시키고,
   막힌 지점(빌드/런타임 에러)을 `reports/.../02-engineering.md`에 목록화.
   *완료기준*: 최소 1개 서류군에서 입력→DOCX 성공, 실패 항목은 재현 절차 포함 기록.
2. **[P1] 로컬 구동·빌드 점검** — `.env.local` 키 유무 확인(ANTHROPIC/SUPABASE),
   `npm run build` 성공 여부 기록. *완료기준*: 빌드 성공/실패 + 누락 키 목록.
   *주의*: 검증 전 코드 임의 프로덕션 배포 금지.

### 디자인팀(T3)
1. **[P1] 위저드·미리보기·계약목록 UX 현황 진단** — `components/trust/{Wizard,ContractsView}.tsx`,
   `steps/*` 흐름을 기준으로 단계 누락·혼선·미리보기 가독성 이슈를 화면 단위로 3~5건 도출.
   *완료기준*: 이슈 목록(화면·문제·개선안) + 우선순위 표기.
2. **[P2] 디자인 방향 한 줄 정의** — "신탁 실무자가 신뢰하는 서류 도구" 톤(전문·정확·신뢰)
   기준 컬러/타이포 방향 초안. *완료기준*: 1페이지 방향 메모(브랜드명 미정 전제).

### 마케팅팀(T4)
1. **[P2] 세일즈 원페이저 골격 초안** — 타깃 4세그먼트(신탁사·시행사·시공사·증권사)별
   페인포인트→가치제안 1줄씩. *완료기준*: 원페이저 아웃라인(섹션·핵심문구).
   *주의*: "법률 자문"이 아닌 "서류 작성·정보 제공 도구"로 표현(변호사법 경계).
2. **[P2] 가격/패키지 방향 가설** — B2B 멀티테넌트 구독 전제, 경쟁/대체 수단 대비
   과금 축(서류 건수/시트 수/조직) 가설 2~3안. *완료기준*: 가설 비교표(결정은 대표님).

### 사업팀(T5 · 신탁 전문)
1. **[P0] 1차 자동화 대상 계약서 정의서** — 신탁 실무 빈도·반복성 기준으로
   1차 대상 계약서 1~2종 선정 + 필수조항·필수입력필드 목록화.
   *완료기준*: 계약서별 [구조·필수조항·필드·verbatim 원본 출처] 정의서.
   *주의*: 조문은 검증된 원본 출처만 인용(추정 조문 금지).
2. **[P0] 현행 산출 DOCX 조문 정확성 검수 착수** — 개발팀 검증 산출물(P1)이 나오면
   `trust-automatic` 원본과 verbatim 대조 1건 시작. *완료기준*: 대조 체크리스트 + 불일치 목록.

## 3. 로드맵 메모

- **단계 1 (현재 주):** 엔진 동작 검증 + 1차 계약서 정의서 → "무엇을, 어디까지 자동화하나" 확정.
- **단계 2:** 정의서 기반 모델/스키마/조문 확장 + UX 정비 + DOCX verbatim 검수 통과율 측정.
- **단계 3:** 운영 기반(.env/Supabase/배포) + 온보딩 + 마케팅 론칭.
- **의존성:** 개발 엔진검증(P0) ← 사업 정의서(P0)에 일부 의존 / 마케팅·디자인은 병렬 선행 가능.
- **★대표님 결정 필요 (블로커 후보):** ① 브랜드명 확정 ② 배포 환경(Vercel 등) 선택.
  두 건은 단계 3 진입 전 필요 — 단계 1~2는 결정 없이도 진행 가능하므로 일정 영향 최소.
